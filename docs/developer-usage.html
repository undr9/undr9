Quick Start

Start here if you want to run a local node, create a first record, and immediately understand what shape the API accepts.

0. How to create a database

UNDR9 does not currently expose a separate CREATE DATABASE query or SQL-like database creation command. In the current implementation, you create a database in one of two practical ways depending on what you mean by “database.”

New physical database Start UNDR9 with a new empty --root directory. That directory becomes the database storage root.
New logical database / tenant Use namespace-prefixed IDs such as tenant_a:node_1 and tenant_b:node_1 inside the same engine.
Current model: UNDR9 uses one engine per storage root and supports multi-database style separation through namespace prefixes rather than a separate per-database engine.
CLICreate a new physical database
cargo run -p undr9-cli -- serve \
  --root ./data/my_first_db \
  --bind 127.0.0.1:8080 \
  --node-id node-1
JSONCreate a new logical database via namespace
{
  "id": "tenant_a:memory_1",
  "node_type": "Memory",
  "properties": {
    "title": { "kind": "String", "value": "First record in tenant_a" }
  },
  "vectors": {}
}

1. Start UNDR9

CLIStart local server
cargo run -p undr9-cli -- serve \
  --root ./data \
  --bind 127.0.0.1:8080 \
  --node-id node-1
UNDR9 does not terminate TLS in-process. Put Caddy or Traefik in front of it for HTTPS in development or production.

2. Insert a node

A node is the main record type. In UNDR9 a memory is typically represented as a node with memory-specific properties.

HTTPCreate one memory node
curl -X POST http://127.0.0.1:8080/v1/nodes \
  -H 'content-type: application/json' \
  -H 'x-api-key: writer' \
  -d '{
    "id": "tenant_a:memory_1",
    "node_type": "Memory",
    "properties": {
      "title": { "kind": "String", "value": "Meeting summary" },
      "timestamp": { "kind": "Integer", "value": 1710000000000 },
      "importance": { "kind": "Float", "value": 0.82 },
      "confidence": { "kind": "Float", "value": 0.91 }
    },
    "vectors": {
      "default": [0.22, 0.41, 0.90],
      "memory": [0.22, 0.41, 0.90]
    }
  }'

3. Run a query

HTTPGet node by ID
curl -X POST http://127.0.0.1:8080/v1/query \
  -H 'content-type: application/json' \
  -H 'x-api-key: reader' \
  -d '{
    "GetNodeById": {
      "node_id": "tenant_a:memory_1"
    }
  }'

Database Creation & Tenancy

This section explains how to create databases in the current UNDR9 implementation, how to connect to them, and how separation works between different users or tenants.

Two meanings of “database” in UNDR9

In the current implementation, “database” can mean either a separate physical storage root or a logical tenant namespace inside one running engine. Both are valid, but they provide different isolation properties.

Mode How you create it How you connect Isolation level
Physical database Start UNDR9 with a different --root directory. Connect to the server instance bound to that root. Strong operational separation. Different files on disk.
Logical database / tenant Use namespace-prefixed IDs like tenant_a:... and tenant_b:.... Connect to the same server, but send tenant-scoped IDs. Data-model separation inside one engine. Shared process, shared WAL, shared indexes.

Create and connect to separate physical databases

If you want one user or customer to have a completely separate on-disk database, run a separate UNDR9 instance with a separate storage root. This is the clearest form of separation in the current system.

CLICreate and serve database A
cargo run -p undr9-cli -- serve \
  --root ./data/customer_alpha \
  --bind 127.0.0.1:8080 \
  --node-id alpha-node
CLICreate and serve database B
cargo run -p undr9-cli -- serve \
  --root ./data/customer_beta \
  --bind 127.0.0.1:8081 \
  --node-id beta-node

Each server is then connected to through its own base URL.

PythonConnect to database A
from undr9.client import SyncUndr9Client

alpha = SyncUndr9Client(
    base_url="http://127.0.0.1:8080",
    api_key="writer",
)
PythonConnect to database B
from undr9.client import SyncUndr9Client

beta = SyncUndr9Client(
    base_url="http://127.0.0.1:8081",
    api_key="writer",
)
Physical separation means ./data/customer_alpha and ./data/customer_beta have different node, edge, vector, index, and WAL files. Data written to one does not appear in the other.
ProofSame ID in two physical databases
# Write to database A at :8080
POST /v1/nodes
{
  "id": "claim:1001",
  "node_type": "Claim",
  "properties": {
    "owner": { "kind": "String", "value": "customer_alpha" }
  },
  "vectors": {}
}

# Write to database B at :8081
POST /v1/nodes
{
  "id": "claim:1001",
  "node_type": "Claim",
  "properties": {
    "owner": { "kind": "String", "value": "customer_beta" }
  },
  "vectors": {}
}

# These are separate records in separate physical databases because they live in different roots and different running instances.

Create and connect to logical databases with namespaces

If you want many users or tenants to share one UNDR9 engine, create logical databases by prefixing IDs with a namespace. This keeps data grouped by tenant without duplicating the engine.

JSONTenant A records
{
  "id": "tenant_a:claim:1001",
  "node_type": "Claim",
  "properties": {
    "owner": { "kind": "String", "value": "tenant_a" },
    "timestamp": { "kind": "Integer", "value": 1753000000000 }
  },
  "vectors": {}
}

{
  "id": "tenant_a:icd:E11",
  "node_type": "ICD10",
  "properties": {
    "name": { "kind": "String", "value": "Type 2 Diabetes Mellitus" }
  },
  "vectors": {}
}
JSONTenant B records
{
  "id": "tenant_b:claim:1001",
  "node_type": "Claim",
  "properties": {
    "owner": { "kind": "String", "value": "tenant_b" },
    "timestamp": { "kind": "Integer", "value": 1753000000000 }
  },
  "vectors": {}
}

{
  "id": "tenant_b:icd:E11",
  "node_type": "ICD10",
  "properties": {
    "name": { "kind": "String", "value": "Type 2 Diabetes Mellitus" }
  },
  "vectors": {}
}

Connection in this model uses the same base URL because there is only one running engine.

PythonConnect once, address tenants by namespace
from undr9.client import SyncUndr9Client

client = SyncUndr9Client(
    base_url="http://127.0.0.1:8080",
    api_key="writer",
)

# Tenant A
client.get_node("tenant_a:claim:1001")

# Tenant B
client.get_node("tenant_b:claim:1001")
In namespace mode, you do not connect to a different server per tenant. You connect to the same server and choose the tenant by the IDs you use.

How separation works between different users’ databases

The separation model depends on which database style you use. Physical databases and logical databases are not the same thing operationally.

Question Physical database Logical database / namespace
Separate files on disk? Yes No
Separate WAL? Yes No
Separate indexes? Yes No
Separate process / bind address? Usually yes No
Separate by ID prefix? Not required Yes
Important: namespace separation is a data-model separation strategy, not a full security boundary by itself. If you want strict user-to-user isolation today, prefer separate physical databases or enforce namespace access at the application or reverse-proxy layer.

Actual multi-user examples

Example 1Strict separation for two customers
# Customer Alpha
cargo run -p undr9-cli -- serve --root ./data/customer_alpha --bind 127.0.0.1:8080 --node-id alpha

# Customer Beta
cargo run -p undr9-cli -- serve --root ./data/customer_beta --bind 127.0.0.1:8081 --node-id beta

# Connect separately
alpha_client -> http://127.0.0.1:8080
beta_client  -> http://127.0.0.1:8081
Example 2Shared engine with tenant prefixes
# One UNDR9 process
cargo run -p undr9-cli -- serve --root ./data/shared --bind 127.0.0.1:8080 --node-id shared

# Tenant A records
tenant_a:claim:1001
tenant_a:tweet:1880001

# Tenant B records
tenant_b:claim:1001
tenant_b:tweet:1880001

# Same server, tenant chosen by ID prefix

Recommended rule of thumb

  • Use a separate --root when you need strong operational isolation between customers or environments.
  • Use namespaces when you want one shared engine with lower operational overhead and tenant grouping by ID prefix.
  • Use application-level or proxy-level access control if namespace-based tenants should not see one another’s data.

How API keys work today

API keys in the current implementation are authentication and authorization credentials. They do not create databases and they do not choose which database to connect to. They only determine what the caller is allowed to do against a given running UNDR9 server.

Key type Purpose Typical scope
admin_api_key Administrative and maintenance operations Compaction, backup, restore, rebuild indexes, replication, cluster operations
writer_api_key Write and read workloads Create nodes, create edges, update data, run normal application writes
reader_api_key Read-only access Queries, traversal, retrieval, lookups, metrics reads where allowed by deployment

How to create API keys

There is no runtime CreateApiKey endpoint yet. Today, API keys are created by configuring the UNDR9 server before startup. The current default configuration contains three keys:

DefaultsCurrent development keys
admin_api_key  = undr9-dev-admin-key
writer_api_key = undr9-dev-writer-key
reader_api_key = undr9-dev-reader-key

In code, these live in the server configuration under AuthConfig. If you want different keys, change the config values used by the server before launching it.

Rust configAuth configuration shape
pub struct AuthConfig {
    pub enabled: bool,
    pub bootstrap_admin_username: String,
    pub admin_api_key: String,
    pub writer_api_key: String,
    pub reader_api_key: String,
}

Do all API keys connect to the same database?

Yes, if the keys are used against the same running UNDR9 server instance, they all connect to the same database behind that server. The database is selected by the server instance and its storage root, not by the API key.

Same DBThree keys, one database
# One server
cargo run -p undr9-cli -- serve \
  --root ./data/customer_alpha \
  --bind 127.0.0.1:8080 \
  --node-id alpha-node

# All of these talk to the same database at ./data/customer_alpha
x-api-key: undr9-dev-admin-key
x-api-key: undr9-dev-writer-key
x-api-key: undr9-dev-reader-key
Different DBsDifferent server instances
# Database A
http://127.0.0.1:8080  -> ./data/customer_alpha

# Database B
http://127.0.0.1:8081  -> ./data/customer_beta

# The base URL / running instance changes the database.
# The API key changes permissions.

Concepts

This section answers the important “what does this field mean?” questions directly.

Current query language

UNDR9 currently uses a custom JSON query DSL over HTTP. You do not write Cypher, SQL, or GraphQL. Instead, you send a JSON object whose top-level key selects the query kind, for example GetNodeById, Traverse, or RankedRetrieval.

What UNDR9 stores

Node
The primary graph record. A node has an id, a node_type, optional properties, and optional named vectors.
Edge
A directed relationship between two nodes. An edge has its own id, source, target, edge_type, and optional properties.
Memory
A memory is usually a node whose properties include values such as timestamp, importance, and confidence.
Timeline
The temporal dimension of a node, usually driven by a property such as timestamp in epoch milliseconds.
Vector
A semantic representation stored in the node API model but persisted separately from node segment pages in storage.

What “database” means in UNDR9 today

The word “database” can mean two different things in practice, and that is important to understand when operating UNDR9.

Physical database
A physical database is a storage root on disk. If you launch UNDR9 with --root ./data/customer_a, that directory becomes one standalone persisted database with its own nodes, edges, vectors, indexes, and WAL files.
Logical database
A logical database is a namespace inside one physical database. For example, tenant_a:claim:1001 and tenant_b:claim:1001 can coexist inside the same engine and remain separated by ID prefix.

# Physical database

Field reference: what each field means

Field Meaning Accepted values
node_type The logical label or class of a node. Any non-empty string. Examples: Memory, Claim, ICD10, Medicine, Tweet, TwitterUser.
properties Key/value metadata attached to a node or edge. A JSON object whose keys are non-empty strings and whose values use the { "kind": ..., "value": ... } envelope.
timestamp The primary temporal field used by time-range search and memory recency. Recommended: epoch milliseconds as Integer. Engine rule: must be integer-compatible.
kind The type tag inside a property envelope. String, Integer, Float, Boolean, StringList, FloatList.
importance How important a node is for retrieval ranking. Recommended: 0.0 to 1.0. Engine rule: any float-compatible numeric value.
confidence How reliable or trusted the node or relation is. Recommended: 0.0 to 1.0. Engine rule: any float-compatible numeric value.
vectors A map of vector-space names to embedding arrays. A JSON object where each key is a vector name and each value is an array of floats.
vectors.default The canonical fallback vector used when the caller does not specify a named vector space. Recommended for every vector-enabled node.
vectors.memory, vectors.code, vectors.tweet_text Optional named vector slots for different semantic spaces. Any string key is accepted today. Recommended names are human-readable and stable.

Property kinds

Kind JSON example Use case
String { "kind": "String", "value": "Metformin" } Text fields such as titles, usernames, labels, and codes.
Integer { "kind": "Integer", "value": 1753000000000 } Timestamps, ages, counts, lengths of stay, quantities.
Float { "kind": "Float", "value": 0.92 } Importance, confidence, similarity scores.
Boolean { "kind": "Boolean", "value": true } Flags and yes/no state markers.
StringList { "kind": "StringList", "value": ["graph", "search"] } Tags, hashtags, labels, aliases.
FloatList { "kind": "FloatList", "value": [0.1, 0.2, 0.3] } Legacy embedding property format. New code should prefer vectors.

Recommended conventions versus engine rules

node_type Engine requires a non-empty string. The docs recommend clear singular class names such as Claim or Tweet.
timestamp Engine accepts integer-compatible numeric values. The docs recommend epoch milliseconds everywhere.
importance / confidence Engine accepts integer or float values. The docs recommend normalized float values in the range 0.0 to 1.0.
vectors Engine accepts arbitrary named vector slots. The docs recommend always supplying default if a node has semantic embeddings.

API Reference

This section lists every current query type and gives two examples for each one, plus the fields you can pass.

Query catalog

Query Purpose
GetNodeByIdExact lookup by node ID.
GetNodeByUniqueKeyExact lookup by the unique_key property.
ListNeighborsDirect one-step neighbor inspection.
TraverseBounded graph traversal with optional constraints.
ShortestPathShortest unweighted path using bidirectional BFS.
SearchByLabelFind nodes by node_type / label.
TimeRangeFind nodes in a time window based on a field such as timestamp.
VectorSearchSemantic similarity search using a vector query.
RankedRetrievalHybrid retrieval that combines graph, semantic, temporal, and metadata ranking.

Shared request rules

  • All queries are sent to POST /v1/query.
  • The top-level JSON key must be exactly one query variant name.
  • Traversal-style queries should send explicit limit, max_hops or max_depth, and timeout_ms.
  • Direction values currently used in the public API are Outgoing and Incoming.

timeout_msOptional timeout in milliseconds.
constraints.edge_typesOptional list of allowed edge types.
constraints.node_labelsOptional list of allowed node labels / node_type values.

Architecture

This section summarizes the implementation choices that matter when you are designing schemas and queries.

Storage layout

Path Purpose Format
data/nodes/Node segment snapshotsrkyv
data/edges/Edge segment snapshotsrkyv
data/vectors/Vector segment snapshotsrkyv
data/wal/Durable write-ahead logPostcard framed records
data/indexes/Derived graph indexessnapshot metadata

Traversal and ranking

  • Outgoing and incoming adjacency views are both stored for fast traversal.
  • Shortest path uses bidirectional BFS because edges do not yet carry routing cost weights.
  • Hybrid ranking uses a named retrieval profile instead of scattering hardcoded weights.

Troubleshooting

Use these notes when something does not behave as expected.

I do not know what values are accepted

Start with the field reference in the Concepts section. The short version is: node_type is any non-empty string, property kind must be one of six supported values, timestamp should be an integer epoch millisecond value, and vectors are maps of names to float arrays.

My query returns no nodes

  • Check whether the starting node exists.
  • Check whether the direction is correct: Outgoing versus Incoming.
  • Check that your edge_type and constraints.edge_types are not contradictory.
  • Check that your node_labels match real node_type values.

My Python example crashes with connection refused

That means UNDR9 is not listening at the target base_url. Start the server first.

CLIStart server before examples
cargo run -p undr9-cli -- serve \
  --root ./data \
  --bind 127.0.0.1:8080 \
  --node-id node-1
