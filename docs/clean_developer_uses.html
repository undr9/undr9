UNDR9 Developer Documentation

Graph-memory database documentation for developers.

UNDR9 is a self-hosted Rust database that combines graph relationships, temporal retrieval, vector similarity, and explainable ranked retrieval. This page is a clean, developer-first reference for the database engine and its architecture, modeled after standard documentation sites with a clear index and sectioned reading flow.

Storage Model

Append-oriented, WAL-backed, snapshot-published, single-node first.

Query Model

Typed JSON DSL with lookup, traversal, shortest path, temporal, vector, and ranked plans.

Retrieval Model

Structure + semantics + time + importance + confidence with explicit score breakdowns.

Introduction

UNDR9 is an open-source, self-hosted graph-memory database written in Rust. The project is organized as a layered multi-crate workspace so that storage, WAL, indexing, query planning, ranking, API transport, and operations remain explicit and independently evolvable.

At a practical level, UNDR9 stores nodes and edges as the canonical data model, then layers retrieval semantics on top through reserved metadata such as timestamps, importance, confidence, and vectors. The result is a database suited for applications that need graph relationships, semantic similarity, recency-aware memory retrieval, and explainable ranking.

How to read this guide
Start with Quick Start and Core Concepts, then move into Query Language, Memory Operations, and Internals for a deeper understanding of how the engine works.

Quick Start

The fastest way to understand UNDR9 is to boot a local node, create a first node, and issue a query through the HTTP API.

1. Start a local database node

cargo run -p undr9-cli -- serve --root ./data --bind 127.0.0.1:8080 --node-id node-1

2. Create a node

curl -X POST http://127.0.0.1:8080/v1/nodes \
  -H 'Content-Type: application/json' \
  -H 'x-api-key: undr9-dev-writer-key' \
  -d '{
    "id": "memory:welcome",
    "node_type": "Memory",
    "properties": {
      "title": { "kind": "String", "value": "Welcome to UNDR9" },
      "timestamp": { "kind": "Integer", "value": 1753000000000 },
      "importance": { "kind": "Float", "value": 0.9 },
      "confidence": { "kind": "Float", "value": 1.0 }
    },
    "vectors": {
      "default": [0.24, 0.12, 0.88, 0.39]
    }
  }'

3. Query the node

curl -X POST http://127.0.0.1:8080/v1/query \
  -H 'Content-Type: application/json' \
  -H 'x-api-key: undr9-dev-reader-key' \
  -d '{
    "GetNodeById": {
      "node_id": "memory:welcome"
    }
  }'

4. Think in two database layers

  • Physical database: the storage root passed through --root.
  • Logical database: namespace-prefixed IDs inside the same engine, such as tenant_a:claim:1001.

Core Concepts

UNDR9 is easier to use when its core ideas are explicit from the beginning.

Concept Meaning
Node The primary record type. Nodes carry domain data and optional memory metadata.
Edge A typed relationship between two nodes, with optional properties.
Property A typed key-value field. Values are tagged as String, Integer, Float, Boolean, StringList, or FloatList.
Vector A named embedding slot stored under vectors, typically with default as the primary semantic vector.
Namespace A logical database boundary modeled by prefixing IDs with a namespace segment.
PlanKind The query planner result that determines how a request is executed.

Single-node first architecture

The repository roadmap is intentionally sequential: durable storage and recovery first, query and API next, then retrieval, operational hardening, transactions, and later replication or clustering. This makes the engine easier to reason about and keeps the durability model clear.

Typed JSON contract

UNDR9 uses a typed HTTP/JSON interface. Properties are tagged with explicit kinds, and queries are top-level JSON variants instead of SQL or GraphQL strings. This keeps the database behavior deterministic and easy to validate.

Memory Model

UNDR9 does not introduce a separate persisted memory primitive in V1. Instead, memory is modeled as Node + Metadata.

Reserved memory metadata

Field Type Role
embedding FloatList Semantic similarity input for retrieval.
timestamp Integer UTC epoch milliseconds used for temporal search and freshness scoring.
importance Float Normalized application-provided importance signal.
confidence Float Normalized confidence or trust signal.

What temporal memory means

A temporal memory in UNDR9 is simply a node with a meaningful timestamp. That timestamp makes the node eligible for temporal range queries and time-aware ranking inside ranked retrieval.

Important
Applications can store any additional domain properties they need, but the reserved memory fields should keep their documented meanings and types.

Query Language

UNDR9 exposes a JSON DSL through POST /v1/query. The database planner parses the request into a typed AST and executes it against a graph snapshot and derived indexes.

Supported query families

  • GetNodeById and GetNodeByUniqueKey for exact lookups.
  • ListNeighbors, Traverse, and ShortestPath for graph exploration.
  • SearchByLabel for node-type searches.
  • TimeRange for temporal search.
  • VectorSearch for semantic retrieval.
  • RankedRetrieval for hybrid ranking.

Build the graph first

Query examples should reference a graph that was actually created. In a personal assistant memory graph, do not use free-form sentence fragments as node identifiers. Create stable entity IDs first, then connect them with typed edges.

Node ID design rule
A node id should be a stable identity key, not natural language. Good ids look like user_me, goal_weight_loss_2026, or a UUID. The sentence itself belongs in properties such as summary, goal_text, or display_name.

GetNodeById

Fetch one exact node by the stable node id you created.

{
  "GetNodeById": {
    "node_id": "goal_weight_loss_2026"
  }
}

GetNodeByUniqueKey

Fetch one exact node through the unique_key property when that is your external lookup key.

{
  "GetNodeByUniqueKey": {
    "unique_key": "goal:weight_loss:2026"
  }
}

ListNeighbors

Inspect the direct facts attached to one node. This example asks for everything directly connected to Sarah.

{
  "ListNeighbors": {
    "node_id": "person_sarah"
  }
}

Traverse

Walk outward through the graph. This example starts from the user and collects the goal plus the target date.

{
  "Traverse": {
    "start_node_id": "user_me",
    "direction": "Outgoing",
    "max_hops": 2,
    "limit": 20,
    "timeout_ms": 5000,
    "constraints": {
      "edge_types": ["HAS_GOAL", "TARGET_DATE"],
      "node_labels": ["Goal", "Date"]
    }
  }
}

ShortestPath

Explain how two memories are connected. This example shows the assistant linking the reminder to Sarah.

{
  "ShortestPath": {
    "source_node_id": "reminder_buy_sarah_gift",
    "target_node_id": "person_sarah",
    "direction": "Outgoing",
    "max_depth": 4,
    "limit": 20,
    "timeout_ms": 5000,
    "constraints": {
      "edge_types": ["ABOUT_PERSON"],
      "node_labels": ["Reminder", "Person"]
    }
  }
}

SearchByLabel

List all nodes of one type. This example returns every goal stored for the assistant.

{
  "SearchByLabel": {
    "label": "Goal"
  }
}

TimeRange

Find memories inside a time window. This example returns reminders and other time-stamped memories for 2026.

{
  "TimeRange": {
    "field": "timestamp",
    "from_epoch_ms": 1767225600000,
    "to_epoch_ms": 1798675200000,
    "limit": 20
  }
}

VectorSearch

Run pure semantic retrieval. This example looks for memories related to health goals and weight loss.

{
  "VectorSearch": {
    "query_vector": [0.91, 0.22, 0.15, 0.78],
    "node_type": "Goal",
    "limit": 5
  }
}

RankedRetrieval

Run hybrid retrieval. This example asks for the best memories related to Sarah by combining structure, semantics, and recency.

{
  "RankedRetrieval": {
    "query_vector": [0.44, 0.81, 0.13, 0.62],
    "reference_node_id": "person_sarah",
    "edge_type": "ABOUT_PERSON",
    "from_epoch_ms": 1767225600000,
    "to_epoch_ms": 1798675200000,
    "limit": 5,
    "now_epoch_ms": 1775000000000,
    "retrieval_profile": "v1-default"
  }
}

Validation rules

  • Vectors must be non-empty when supplied.
  • Limits and bounds must be positive where required.
  • RankedRetrieval requires at least a query vector or a reference node.
  • Traversal and shortest path are intentionally bounded by depth, limit, and timeout.

Memory Operations

In practice, memory operations in UNDR9 are graph writes and reads with retrieval-aware metadata.

Create and update

  • Create or upsert nodes through POST /v1/nodes.
  • Create or upsert edges through POST /v1/edges.
  • Update existing records with PUT endpoints.
  • Delete records with DELETE endpoints.

Transactions

UNDR9 supports explicit snapshot-style transaction sessions. A transaction begins from committed state, stages node or edge writes in memory, allows queries against that staged view, and then either commits atomically or rolls back. 

POST /v1/transactions
POST /v1/transactions/{id}/nodes
POST /v1/transactions/{id}/edges
POST /v1/transactions/{id}/query
POST /v1/transactions/{id}/commit
POST /v1/transactions/{id}/rollback

Conflict model

Transactions use optimistic conflict detection through record lineage revisions. If a record changes after a transaction snapshot is taken, a stale transaction commit fails instead of silently overwriting newer state.

Vector Search

Vector search is a first-class retrieval mode in UNDR9. Vectors are named slots under the node vectors map, and the engine uses normalized cosine similarity for semantic scoring.

Vector search

{
  "VectorSearch": {
    "query_vector": [1.0, 0.0],
    "node_type": "Memory",
    "limit": 5
  }
}

Ranked retrieval

Ranked retrieval is the hybrid path. It combines structural relevance, vector similarity, temporal freshness, importance, and confidence into one score, then returns a score breakdown for every result. 

{
  "RankedRetrieval": {
    "query_vector": [1.0, 0.0],
    "reference_node_id": "tenant_a:claim:1001",
    "edge_type": "HAS_DIAGNOSIS",
    "from_epoch_ms": 1753000000000,
    "to_epoch_ms": 1754000000000,
    "limit": 5,
    "now_epoch_ms": 1754200000000,
    "retrieval_profile": "v1-default"
  }
}

Default ranking weights

  • Structural: 0.30
  • Semantic: 0.30
  • Temporal: 0.15
  • Importance: 0.15
  • Confidence: 0.10

Deployment

The current operational center of gravity is the single-node service. UNDR9 expects reverse-proxy TLS termination and focuses first on durability, maintenance, and predictable local operation.

Recommended single-node shape

  • Reverse proxy such as Caddy or Traefik in front of UNDR9.
  • UNDR9 node running the local storage root.
  • Periodic maintenance through CLI or admin endpoints.

Operational commands

cargo run -p undr9-cli -- verify-storage --root ./data
cargo run -p undr9-cli -- compact-storage --root ./data
cargo run -p undr9-cli -- rebuild-indexes --root ./data
cargo run -p undr9-cli -- backup-storage --root ./data --destination ./backup
cargo run -p undr9-cli -- restore-storage --root ./data --source ./backup
cargo run -p undr9-cli -- repair-storage --root ./data

Container deployment

Docker artifacts are provided for local and operational deployment. The current documentation also includes a docker compose path for bringing up the service with persistent volumes.

Performance

UNDR9 performance is shaped by bounded execution, append-oriented durability, and rebuildable derived indexes.

What the engine optimizes for

  • Durable writes with a clear WAL and snapshot pipeline.
  • Index-backed reads instead of unbounded graph scans.
  • Bounded traversal and shortest-path exploration.
  • Controlled candidate generation before hybrid ranking.

Derived index strategy

Exact lookup, adjacency, label/type, temporal, and vector candidate structures are derived from canonical state. This keeps the storage model simple while still enabling fast reads.

Benchmark focus

  • CRUD latency and throughput
  • Traversal and shortest-path workloads
  • Temporal range retrieval
  • Vector similarity retrieval
  • Hybrid ranked retrieval
  • Recovery, replay, and maintenance operations

Internals

UNDR9 is easiest to understand as a set of clear subsystem layers rather than a single monolith.

Layer Role
Storage Owns manifest, snapshots, WAL application, lineage tracking, and transaction sessions.
WAL Provides durable ordered mutation logging with framed records and replay.
Index Builds derived read structures such as adjacency, temporal index, and vector candidates.
Query Defines the AST, planner, executor, and result shapes for all query types.
Memory Implements ranking math, semantic similarity, temporal freshness, and consolidation semantics.
API Exposes HTTP endpoints and coordinates storage, index rebuilds, and query snapshots.
Operations Handles serve, verify, compact, rebuild, repair, backup, and restore workflows.

Write path

  1. Validate the incoming write batch.
  2. Append the batch to the WAL.
  3. Update in-memory node and edge state.
  4. Publish new snapshots and manifest metadata.
  5. Rebuild derived indexes.

Read path

  1. Take a graph snapshot from committed state.
  2. Plan the JSON query into a typed execution plan.
  3. Execute against indexes and graph records.
  4. Return nodes, edges, ranked results, or path information depending on the plan kind.

FAQ

Is UNDR9 a graph database or a memory database?

It is both in practice. The canonical storage model is graph-based, while retrieval semantics treat certain node metadata as memory-oriented signals.

Does UNDR9 generate embeddings?

No. UNDR9 stores and uses vectors, but embedding generation is expected to happen outside the database.

What does a logical database mean here?

A logical database is typically represented by namespace-prefixed IDs inside one running engine. The physical database is the storage root on disk.

Why would a query return no nodes?

Typical reasons are wrong IDs, wrong edge or label constraints, empty result windows, or namespace mismatches.

Is namespace separation a security boundary?

No. Namespace separation is primarily a data-model and logical organization mechanism, not a full security boundary on its own.

Roadmap

UNDR9 evolves through explicit milestones. The implementation sequence is itself part of the product design.

Phase Focus
Milestones 0-1 Workspace structure, storage bootstrap, WAL, durable single-node CRUD, and crash recovery.
Milestone 2 Indexes, planner and executor, authenticated CRUD and query API.
Milestone 3 Vector search, temporal range retrieval, ranked retrieval, and retrieval scoring.
Milestone 4 Compaction, backup, restore, repair, verify, rebuild, and richer operational tooling.
Milestone 5 Explicit transaction sessions with snapshot reads and optimistic conflict detection.
Future direction Measured replication, cluster coordination, broader distributed readiness, and continued retrieval evolution.
